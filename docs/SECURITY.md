# セキュリティ設定ドキュメント

本ドキュメントでは、ROCK CASTINGアプリケーションのセキュリティ設定について説明します。

## 📚 セキュリティの基礎知識

### セキュリティヘッダーとは？

**セキュリティヘッダー**は、Webアプリケーションがブラウザに送る特別な指示です。これにより、ブラウザがどのようにページを表示・処理するかを制御し、様々な攻撃から保護できます。

**例**: 銀行の警備員が「この建物には武器を持ち込めません」と案内するようなものです。

### なぜセキュリティが必要なのか？

Webアプリケーションは、悪意のあるユーザーからの攻撃の標的になる可能性があります。適切なセキュリティ対策を講じることで、以下のような被害を防ぐことができます：

- **個人情報の漏洩**: ユーザーのメールアドレスやパスワードが盗まれる
- **不正アクセス**: 他人のアカウントに勝手にログインされる
- **データの改ざん**: 重要な情報が書き換えられる
- **サービス停止**: 攻撃によってサービスが使えなくなる

## 実装済みセキュリティ機能

### 1. セキュリティヘッダー

`src/middleware.ts`で以下のセキュリティヘッダーを設定しています：

#### X-Content-Type-Options: `nosniff`

**用語説明**: 
- **MIMEタイプ**: ファイルの種類を表す識別子（例: `text/html`, `image/png`）
- **スニッフィング**: ブラウザがファイルの中身を見て、実際の種類を推測すること

**何を防ぐか**: 
悪意のあるユーザーが、JavaScriptファイルを画像ファイルとして偽装してアップロードし、ブラウザに実行させようとする攻撃を防ぎます。

**具体例**: 
悪意のあるユーザーが`malicious.js`というファイルを`image.png`としてアップロード。ブラウザが中身を見て「これはJavaScriptだ」と判断し実行してしまうのを防ぎます。

#### X-Frame-Options: `DENY`

**用語説明**: 
- **フレーム**: Webページの中に別のWebページを埋め込む機能（`<iframe>`タグ）
- **クリックジャッキング**: 透明なフレームを使って、ユーザーに気づかれずに別の操作をさせる攻撃

**何を防ぐか**: 
このアプリケーションのページを他のサイトのフレーム内に埋め込むことを完全に禁止します。

**具体例**: 
悪意のあるサイトが、このアプリのログインページを透明なフレームで覆い、ユーザーが「いいね」ボタンをクリックしたつもりが、実際にはログインボタンをクリックしてしまう攻撃を防ぎます。

#### X-XSS-Protection: `1; mode=block`

**用語説明**: 
- **XSS（Cross-Site Scripting）**: 悪意のあるJavaScriptコードをWebページに注入する攻撃
- **クロスサイト**: 別のサイトから来たスクリプトを実行すること

**何を防ぐか**: 
ブラウザのXSS検出機能を有効にし、XSS攻撃を検出した場合はページの読み込みをブロックします。

**具体例**: 
悪意のあるユーザーがコメント欄に`<script>alert('攻撃')</script>`と入力し、他のユーザーがそのページを見たときに悪意のあるコードが実行されるのを防ぎます。

#### Strict-Transport-Security（HSTS）

**用語説明**: 
- **HTTPS**: 暗号化された安全な通信プロトコル
- **HTTP**: 暗号化されていない通信プロトコル（危険）
- **HSTS（HTTP Strict Transport Security）**: ブラウザに「このサイトには必ずHTTPSで接続しろ」と指示する仕組み

**何を防ぐか**: 
一度HTTPSで接続した後は、ブラウザが自動的にHTTPS接続を強制します。これにより、中間者攻撃（通信を盗聴・改ざんする攻撃）を防ぎます。

**具体例**: 
ユーザーが`http://example.com`と入力しても、ブラウザが自動的に`https://example.com`にリダイレクトし、安全な接続を保証します。

#### Referrer-Policy: `strict-origin-when-cross-origin`

**用語説明**: 
- **Referrer（リファラー）**: ユーザーがどのページから来たかを示す情報
- **オリジン**: プロトコル（http/https）、ドメイン、ポート番号の組み合わせ

**何を防ぐか**: 
外部サイトにリンクする際に、どのページから来たかの情報を適切に制限します。プライバシーを保護し、機密情報が漏れるのを防ぎます。

**具体例**: 
このアプリからGoogleにリンクする場合、Googleには「このアプリから来た」ことは伝えますが、具体的なページURLは伝えません。

#### Permissions-Policy

**用語説明**: 
- **Permissions（権限）**: ブラウザがデバイスの機能（カメラ、マイクなど）にアクセスする許可

**何を防ぐか**: 
このアプリケーションでは、カメラ、マイク、位置情報へのアクセスを完全に禁止します。不要な権限要求を防ぎます。

**具体例**: 
悪意のあるコードが「カメラへのアクセスを許可してください」と要求しても、この設定により要求自体が無効になります。

#### Content-Security-Policy（CSP）

**用語説明**: 
- **CSP**: ブラウザに「どのリソース（スクリプト、画像、スタイルなど）をどこから読み込んでいいか」を指示する仕組み

**何を防ぐか**: 
悪意のあるスクリプトやリソースの読み込みを防ぎます。XSS攻撃の最も効果的な対策の一つです。

**設定内容**: 
- スクリプト: このアプリ自身からのみ読み込み可能
- スタイル: このアプリ自身からのみ読み込み可能（Ant Design用に`unsafe-inline`も許可）
- 画像: このアプリ、データURL、すべてのHTTPSサイトから読み込み可能
- 接続先: SupabaseとStripeのみ許可
- フレーム: Stripeの決済画面のみ許可

**具体例**: 
悪意のあるユーザーが`<script src="http://evil.com/attack.js"></script>`を注入しても、CSPにより読み込みがブロックされます。

### 2. 認証・認可

**用語説明**:
- **認証（Authentication）**: 「あなたは誰ですか？」を確認すること（ログイン）
- **認可（Authorization）**: 「あなたはこの操作をしてもいいですか？」を確認すること（権限チェック）
- **トークン**: 認証済みであることを証明する文字列（例: 入場券のようなもの）
- **ミドルウェア**: リクエストが処理される前に実行される処理

#### tRPC認証ミドルウェア

`src/server/trpc.ts`の`protectedProcedure`でSupabase認証トークンを検証：

**処理の流れ**:
1. リクエストヘッダーから`Authorization: Bearer <token>`を取得
   - **Bearer**: 「このトークンを持っている人が送ってきました」という意味
2. Supabaseでトークンを検証
   - トークンが有効か、期限切れでないかを確認
3. ユーザー情報をPrismaから取得してコンテキストに追加
   - データベースからユーザー情報を取得
4. 認証失敗時は`UNAUTHORIZED`エラーを返却
   - トークンが無効な場合、エラーを返して処理を中断

**何を防ぐか**: 
ログインしていないユーザーや、無効なトークンを持つユーザーが、保護された機能（プロフィール編集、メッセージ送信など）にアクセスするのを防ぎます。

**具体例**: 
ログインせずに「プロフィールを編集」ボタンを押しても、認証ミドルウェアが「ログインしてください」というエラーを返します。

#### クライアント側の認証トークン送信

`src/lib/trpc/Provider.tsx`でSupabaseセッションからトークンを取得し、tRPCリクエストのヘッダーに自動的に含めます。

**処理の流れ**:
1. ユーザーがログインすると、Supabaseがトークンを発行
2. そのトークンがブラウザに保存される
3. APIリクエストを送る際に、自動的にトークンをヘッダーに含める
4. サーバー側でトークンを検証

**何を防ぐか**: 
ユーザーが手動でトークンを送信する必要がなくなり、セキュリティホールが生まれるリスクを減らします。

### 3. レート制限

**用語説明**:
- **レート制限**: 一定時間内に送信できるリクエストの数を制限すること
- **APIエンドポイント**: サーバーが提供する機能への入り口（例: `/api/login`）

**何を防ぐか**: 
悪意のあるユーザーが短時間に大量のリクエストを送って、サーバーをダウンさせたり（DDoS攻撃）、パスワードを総当たりで試したり（ブルートフォース攻撃）するのを防ぎます。

**設定内容**:
- **制限**: 1分間に100リクエストまで
  - これを超えると「429 Too Many Requests」エラーを返す
- **ストレージ**: メモリ内のMap（本番環境ではRedisなどの外部ストレージを推奨）
  - **Map**: JavaScriptのデータ構造の一種
  - **Redis**: 高速なデータベース（本番環境ではこちらを使うとより堅牢）
- **レスポンスヘッダー**: 
  - `X-RateLimit-Limit`: 制限値（100）
  - `X-RateLimit-Remaining`: 残りリクエスト数
  - `X-RateLimit-Reset`: 制限がリセットされる時刻

**具体例**: 
悪意のあるユーザーが1秒間に1000回ログインを試みても、1分間に100回までしか試行できません。これにより、パスワードの総当たり攻撃を防ぎます。

### 4. CORS設定

**用語説明**:
- **CORS（Cross-Origin Resource Sharing）**: 異なるオリジン（ドメイン）間でリソースを共有する仕組み
- **オリジン**: プロトコル、ドメイン、ポート番号の組み合わせ（例: `https://example.com:3000`）
- **クロスオリジン**: 異なるオリジンからのリクエスト

**何を防ぐか**: 
悪意のあるサイトが、このアプリのAPIを勝手に呼び出して、ユーザーの情報を盗んだり、不正な操作を行ったりするのを防ぎます。

**設定内容**:
- **許可オリジン**: 
  - `NEXT_PUBLIC_APP_URL`（このアプリの本番URL）
  - `localhost:3000`（開発環境）
  - VercelのデプロイURL
  - これら以外のオリジンからのリクエストは拒否される
- **認証情報の送信を許可**: `Access-Control-Allow-Credentials: true`
  - クッキーや認証トークンを送信できるようにする
- **許可メソッド**: GET, POST, PUT, DELETE, OPTIONS
  - これらのHTTPメソッドのみ許可

**具体例**: 
悪意のあるサイト`evil.com`が、このアプリのAPIを呼び出そうとしても、CORS設定によりブロックされます。これにより、ユーザーが`evil.com`を開いている間に、このアプリのデータが盗まれるのを防ぎます。

### 5. 環境変数検証

**用語説明**:
- **環境変数**: アプリケーションの設定を保存する変数（パスワードやAPIキーなど）
- **検証**: 必要な設定が正しく設定されているか確認すること

**何を防ぐか**: 
重要な設定（データベースのパスワード、APIキーなど）が設定されていない状態でアプリを起動すると、予期しない動作やセキュリティホールが生まれる可能性があります。起動時にチェックすることで、問題を早期に発見できます。

**設定内容**:
- アプリケーション起動時に必須環境変数をチェック
- 本番環境では未設定の場合にエラーを投げる（アプリが起動しない）
- 開発環境では警告のみ（開発を続けられる）

**必須環境変数とその役割**:
- `NEXT_PUBLIC_SUPABASE_URL`: Supabase（認証・データベースサービス）のURL
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`: Supabaseの公開キー（誰でも見えるが、権限は限定的）
- `SUPABASE_SERVICE_ROLE_KEY`: Supabaseの管理者キー（機密情報、絶対に漏らしてはいけない）
- `DATABASE_URL`: データベースへの接続情報（パスワードを含む）
- `STRIPE_SECRET_KEY`: Stripe（決済サービス）の秘密キー
- `STRIPE_WEBHOOK_SECRET`: Stripeからの通知を検証するための秘密キー
- `RESEND_API_KEY`: Resend（メール送信サービス）のAPIキー
- `RESEND_FROM_EMAIL`: メール送信元のアドレス

**具体例**: 
`DATABASE_URL`が設定されていないと、データベースに接続できず、アプリが正常に動作しません。起動時にチェックすることで、すぐに問題を発見できます。

### 6. エラーハンドリング

**用語説明**:
- **エラーハンドリング**: エラーが発生した際の適切な処理
- **スタックトレース**: エラーが発生した場所と、そこに至るまでの関数呼び出しの履歴
- **ログ**: アプリケーションの動作記録

**何を防ぐか**: 
エラーメッセージに含まれる詳細な情報（ファイルパス、データベース構造など）が悪意のあるユーザーに漏れると、攻撃の手がかりになってしまいます。適切なエラーハンドリングにより、必要な情報だけをユーザーに表示し、詳細はログに記録します。

#### tRPCエラーハンドリング

`src/server/trpc.ts`の`errorFormatter`で：

**設定内容**:
- **本番環境では詳細なエラー情報を隠す**
  - ユーザーには「サーバーエラーが発生しました」という一般的なメッセージを表示
  - 開発者向けの詳細情報は表示しない
- **スタックトレースを本番環境では返さない**
  - スタックトレースにはファイルパスやコードの構造が含まれるため、攻撃の手がかりになる
- **Zodエラーの詳細を本番環境では返さない**
  - Zodは入力データの検証ライブラリ
  - エラーの詳細が漏れると、データベースの構造が推測される可能性がある

**具体例**: 
開発環境では「`users.email`は必須です」という詳細なエラーが表示されますが、本番環境では「入力内容に誤りがあります」という一般的なメッセージが表示されます。

#### APIエラーログ

`src/app/api/trpc/[trpc]/route.ts`で：

**設定内容**:
- **本番環境では簡潔なログのみ出力**
  - ログファイルが大きくなりすぎるのを防ぐ
  - 重要な情報だけを記録
- **セキュリティ関連のエラーは必ずログに記録**
  - `UNAUTHORIZED`: 認証失敗（ログインに失敗した）
  - `FORBIDDEN`: 認可失敗（権限がない操作を試みた）
  - これらは攻撃の可能性があるため、必ず記録する
- **クライアントIPアドレスをログに記録**
  - 攻撃元を特定するために必要

**具体例**:

### 7. Stripe Webhookセキュリティ

**用語説明**:
- **Webhook**: 外部サービス（この場合はStripe）から、イベントが発生したときに送られてくる通知
- **署名**: データが改ざんされていないことを証明する暗号学的な仕組み
- **IPアドレス**: インターネット上のコンピュータを識別する番号（例: `192.168.1.1`）

**何を防ぐか**: 
悪意のあるユーザーが「決済が完了しました」という偽の通知を送って、実際には決済していないのに決済完了として処理されるのを防ぎます。

**設定内容**:

#### 署名検証

**説明**: 
Stripeが送ってくる通知には、Stripeだけが知っている秘密キーで作成された署名が含まれています。この署名を検証することで、通知が本当にStripeから来たものかを確認できます。

**処理の流れ**:
1. Stripeが決済完了の通知を送る
2. 通知には署名が含まれている
3. サーバー側で署名を検証
4. 署名が正しければ処理を続行、間違っていれば拒否

**具体例**: 
悪意のあるユーザーが「決済完了」という偽の通知を送っても、署名が正しくないため拒否されます。

#### 環境変数検証

**説明**: 
Stripeの設定（APIキーなど）が正しく設定されているか、アプリ起動時に確認します。

**何を防ぐか**: 
設定が不完全な状態でWebhookを受け取ると、正しく処理できずに問題が発生する可能性があります。

#### IPアドレスチェック（参考実装）

**説明**: 
本番環境では、StripeのIPアドレスからのリクエストのみ許可する設定を追加しています（参考実装）。

**注意**: 
Vercelなどのプロキシ経由の場合、実際のIPアドレスが正確に取得できない可能性があるため、署名検証が主な防御手段となります。

#### エラーハンドリング

**説明**: 
エラーが発生した際に、詳細なエラー情報をクライアントに返さないようにします。

**何を防ぐか**: 
エラーメッセージに含まれる情報（データベースの構造など）が攻撃の手がかりになるのを防ぎます。

### 8. Next.js設定

**用語説明**:
- **Next.js**: このアプリケーションで使用しているWebフレームワーク
- **ソースマップ**: 本番環境の圧縮されたコードを、元のコードにマッピングするファイル
- **console.log**: デバッグ用のログ出力

`next.config.js`で以下の設定を行っています：

#### poweredByHeader: false

**説明**: 
`X-Powered-By`ヘッダーを削除します。このヘッダーは「このアプリはNext.jsで動いています」という情報を公開します。

**何を防ぐか**: 
使用している技術スタックを公開すると、その技術の既知の脆弱性を狙った攻撃を受けやすくなります。情報を隠すことで、攻撃の難易度を上げます。

#### productionBrowserSourceMaps: false

**説明**: 
本番環境でソースマップを無効化します。

**何を防ぐか**: 
ソースマップがあると、本番環境のコードから元のコードを推測できてしまいます。これにより、コードの構造やロジックが漏れるのを防ぎます。

#### removeConsole

**説明**: 
本番環境で`console.log`を削除します（`console.error`と`console.warn`は残します）。

**何を防ぐか**: 
`console.log`に機密情報（パスワード、APIキーなど）が含まれていると、ブラウザの開発者ツールで見られてしまいます。削除することで、情報漏洩を防ぎます。

#### セキュリティヘッダーの追加設定

**説明**: 
Next.jsの設定ファイルでも、追加のセキュリティヘッダーを設定しています。これにより、ミドルウェアで設定したヘッダーに加えて、さらにセキュリティを強化します。

## セキュリティチェックリスト

### 本番環境デプロイ前の確認事項

**用語説明**:
- **デプロイ**: アプリケーションを本番環境に公開すること
- **RLS（Row Level Security）**: データベースの行レベルでのセキュリティ設定（ユーザーごとにアクセスできるデータを制限）
- **バックアップ**: データのコピーを保存すること（データが失われた際の復旧用）
- **HTTPS**: 暗号化された安全な通信プロトコル

以下の項目を確認してください：

- [ ] **すべての環境変数が設定されている**
  - Vercelの環境変数設定画面で確認
  - 必須環境変数がすべて設定されているか確認

- [ ] **Stripe Webhook Secretが正しく設定されている**
  - StripeのダッシュボードでWebhook Secretを取得
  - Vercelの環境変数に設定

- [ ] **SupabaseのRLS（Row Level Security）が適切に設定されている**
  - 各テーブルで、ユーザーが自分のデータのみアクセスできるように設定
  - 他人のデータにアクセスできないことを確認

- [ ] **データベースのバックアップが設定されている**
  - Supabaseのダッシュボードでバックアップ設定を確認
  - 定期的にバックアップが取られているか確認

- [ ] **ログ監視が設定されている（Vercelのログなど）**
  - Vercelのダッシュボードでログを確認できるか確認
  - 異常なアクセスがないか定期的にチェック

- [ ] **セキュリティヘッダーが正しく設定されている（ブラウザの開発者ツールで確認）**
  - ブラウザの開発者ツール（F12）を開く
  - 「Network」タブでリクエストを選択
  - 「Response Headers」でセキュリティヘッダーが設定されているか確認

- [ ] **HTTPSが有効になっている**
  - URLが`https://`で始まっているか確認
  - ブラウザのアドレスバーに鍵マークが表示されているか確認

- [ ] **CSP（Content Security Policy）が適切に設定されている**
  - ブラウザの開発者ツールでCSPヘッダーを確認
  - コンソールにCSP違反のエラーが出ていないか確認

## 今後の改善案

### 1. レート制限の外部ストレージ化

**現状**: 
現在はメモリ内のMapを使用しています。これは開発環境では問題ありませんが、本番環境では以下の問題があります：
- サーバーが再起動すると、レート制限の記録が消える
- 複数のサーバーがある場合、それぞれで別々にカウントされる

**改善案**: 
Redisなどの外部ストレージを使用することで、以下のメリットがあります：
- サーバーが再起動しても記録が残る
- 複数のサーバー間でレート制限を共有できる
- より堅牢なレート制限が可能

**用語説明**:
- **Redis**: 高速なデータベース（メモリ上で動作）

### 2. セキュリティ監視

**説明**: 
セキュリティイベント（認証失敗、レート制限超過など）を監視し、異常なアクセスを検知したらアラートを送る仕組みを構築します。

**具体例**: 
- 1時間に10回以上ログインに失敗したIPアドレスを検知
- 管理者にメールやSlackで通知
- 必要に応じて自動的にIPアドレスをブロック

**用語説明**:
- **アラート**: 異常を検知した際の通知

### 3. WAF（Web Application Firewall）

**用語説明**:
- **WAF**: Webアプリケーションを保護するファイアウォール
- **ファイアウォール**: ネットワークの入り口で不正な通信をブロックする仕組み

**説明**: 
Vercelのセキュリティ機能やCloudflareなどのWAFサービスを使用することで、以下の攻撃を自動的にブロックできます：
- SQLインジェクション攻撃
- XSS攻撃
- DDoS攻撃

**メリット**: 
アプリケーション側で実装する必要がなく、WAFサービスが自動的に保護してくれます。

### 4. 定期的なセキュリティ監査

**説明**: 
定期的にセキュリティチェックを行い、脆弱性を早期に発見します。

**実施内容**:
- **依存関係の脆弱性チェック**: `npm audit`コマンドで、使用しているライブラリに既知の脆弱性がないか確認
- **OWASP Top 10のチェック**: 最も一般的な10種類のセキュリティリスクをチェック

**用語説明**:
- **脆弱性**: セキュリティ上の弱点
- **OWASP**: Webアプリケーションのセキュリティに関する国際的な非営利組織
- **OWASP Top 10**: 最も一般的な10種類のセキュリティリスクのリスト

### 5. 2FA（二要素認証）

**用語説明**:
- **2FA（Two-Factor Authentication）**: パスワードに加えて、別の認証方法（SMSコード、認証アプリなど）も必要とする仕組み

**説明**: 
Supabaseの2FA機能を実装することで、パスワードが漏洩しても、第二の認証がなければログインできなくなります。

**メリット**: 
- パスワードが漏洩しても、第二の認証がなければログインできない
- より強固なセキュリティを実現

**具体例**: 
1. ユーザーがパスワードを入力
2. スマートフォンの認証アプリに6桁のコードが表示される
3. そのコードを入力して初めてログインできる
